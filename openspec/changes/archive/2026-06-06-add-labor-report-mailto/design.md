## Context

労務報告（始業・終業）を会社の Microsoft 365 メールで毎日2回・固定宛先へ送る運用を、Webアプリで省力化する。前提として、M365 管理者によるアプリ登録・API権限付与・管理者同意は得られない。Mac の Outlook アプリから通常のメール送信は可能。したがって、アプリ側は「報告内容の作成」と「Outlook へのメール作成画面の引き渡し」までを担い、最終送信はユーザーが Outlook で行う。本設計の根拠は DESIGN.md に準拠する。

想定利用者は単一ユーザー（本人のみ）。認証・マルチユーザー対応は行わない。データはセンシティブな認証情報を持たず、勤務記録・宛先/テンプレート設定・終業メール本文（ヘッダ/フッタ除く）のみ。実行環境はフロントエンド・バックエンド・DB をすべて Docker コンテナとして `docker-compose` で起動する。

## Goals / Non-Goals

**Goals:**
- 管理者承認なしで日々の始業・終業報告の入力負担を削減する
- 宛先・件名・本文を入力済みにした Outlook 新規メール作成画面を `mailto:` で開く
- 月カレンダーで報告状況を可視化し、未報告日を把握できる
- Outlook 起動失敗時にコピーで手動送信できるフォールバックを備える
- 送信完了は検知できない前提を UI 表現（「メール作成済み」）に正しく反映する

**Non-Goals:**
- Microsoft Graph API（`Mail.Send` / `/me/sendMail`）による自動送信
- SMTP AUTH による直接送信
- 送信済み確認の自動検知、添付ファイル送信、承認ワークフロー、複数ユーザーの勤怠集計
- 認証・ログイン機能（単一ユーザー前提のため不要）
- マルチユーザー対応・ユーザー単位のデータ分離
- 始業メール本文の保存（保存対象は終業メール本文のヘッダ/フッタを除いた部分のみ）

## Decisions

### D1: 送信方式は `mailto:` リンク（Graph API / SMTP を不採用）
- **選択**: Webアプリで本文生成 → `mailto:` で Outlook 起動 → ユーザーが送信。
- **理由**: 管理者承認・API権限・認証情報管理が不要。From は Outlook 既定アカウントに委譲でき、送信ログは通常の Outlook 送信として残る。最終操作がユーザーなので誤送信リスクが低い。
- **代替案**: (a) Graph API 直接送信 → Entra ID アプリ登録・管理者同意・セキュリティ審査が必要で過剰権限。(b) SMTP AUTH → 無効化/MFA環境で利用困難、パスワード管理リスク。いずれも前提に反するため不採用。

### D2: mailto URL 生成は `URLSearchParams` ベースのユーティリティに集約
- **選択**: `buildMailtoUrl({ to, cc, bcc, subject, body })` を単一関数として実装（DESIGN 9.2 準拠）。宛先は `encodeURIComponent` 後にカンマ結合、cc/bcc は空なら付与しない。
- **理由**: エンコード漏れ・空パラメータ付与の防止をテスト可能な1箇所に閉じ込める。仕様（outlook-mail-compose）のシナリオがそのままユニットテストになる。
- **代替案**: 各画面で文字列結合 → エンコード漏れ・重複ロジックの温床。不採用。

### D3: ステータスは記録イベントの単方向前進で管理
- **選択**: `not_started → start_recorded → start_mail_created → end_recorded → end_mail_created`。ステータスは `work_reports` の時刻系カラム（`start_time`・`end_time`・`start_mail_created_at`・`end_mail_created_at`）から導出可能とし、保存値はこれらのタイムスタンプを正とする。
- **理由**: 状態を別カラムで二重管理せず、タイムスタンプから一意に算出することで不整合を防ぐ。
- **代替案**: 独立した `status` カラム → タイムスタンプとの二重管理で不整合リスク。導出を基本とし、必要なら表示用にのみ計算する。

### D4: データモデルは `work_reports` と `mail_settings` の2テーブル（単一ユーザー）
- **選択**: DESIGN 7 のスキーマを基本とし、単一ユーザー前提で user 単位のカラム/スコープは持たない。`work_reports` は `work_date` を一意キー（1日1レコード）として upsert し、`work_style`（enum: `office`/`remote`）と `end_mail_body`（text, nullable）を持つ。`end_mail_body` には終業メール本文のうちヘッダ・フッタを除いた本文部分（＝置換後の本文テンプレート）を保存する。始業メール本文は保存しない（始業用の本文カラムは設けない）。`mail_settings` は単一行（1レコード）で、宛先・始業/終業の件名/本文テンプレートに加え、終業メールの `end_header_template`・`end_footer_template` を保持する。
- **理由**: 終業報告は実績を含み後から参照する価値があるため本文（定型のヘッダ/フッタを除く）を残す。始業報告は定型のみで保存価値が低い。MVP に必要十分。
- **代替案**: 始終業とも本文を保存 → 始業は定型のみで冗長。メール履歴テーブルを別途追加 → 送信検知できない方式では過剰、MVP外。

### D5: 永続化は PostgreSQL + FastAPI、全体を Docker で動作させる
- **選択**: DB は PostgreSQL。バックエンドは FastAPI + SQLAlchemy + Alembic（マイグレーション）。フロントは React + TypeScript + Tailwind + shadcn/ui、カレンダーは FullCalendar または React DayPicker。フロントエンド・バックエンド・PostgreSQL の3コンテナを `docker-compose` で起動する。認証は設けない。
- **理由**: 永続化先を PostgreSQL に固定し、アプリ全体を Docker で動作させる要件。フロントから直接 DB を触らず FastAPI 経由とし、スキーマ管理を Alembic に集約する。`docker-compose` で開発・実行環境を再現可能にする。
- **代替案**: Supabase（BaaS）→ 要件外（PostgreSQL/Docker 指定）。フロントから直接 DB アクセス → セキュリティ/接続管理上不可、バックエンドを挟む。ローカル直接実行（非Docker）→ 要件外。

### D6: 送信完了を「送信済み」と表現しない
- **選択**: UI 文言は「Outlookで始業/終業報告メールを作成」「メール作成済み」。送信後の「Outlookで送信しました」チェックはユーザー自己申告として別管理。
- **理由**: mailto では送信を検知できない（DESIGN 10.1）。誤った完了表現を避ける。

### D7: 終業メールのヘッダ/フッタはテンプレート化、勤務区分は2値固定
- **選択**: 終業メールの定型ヘッダ・フッタを `mail_settings` の `end_header_template`・`end_footer_template` として保存し、設定画面で編集可能にする（低頻度）。終業メール全文 = ヘッダ + 本文テンプレート（変数置換後） + フッタ で構成し、日次保存する `end_mail_body` は中央の本文テンプレート置換結果のみとする。勤務区分 `work_style` は `office`（オフィス）/ `remote`（リモート）の2値固定とする。
- **理由**: ヘッダ/フッタは全レコードで共通かつ稀にしか変わらないため、日次レコードへの重複保存を避け設定として一元管理する。本文を独立テンプレートにすることで、文字列の抽出処理なしに置換結果をそのまま `end_mail_body` に保存できる。勤務区分を2値に固定すると入力UI（トグル/選択）と将来の集計が単純になる。
- **代替案**: 終業本文を単一テンプレートに保持し保存時にヘッダ/フッタを文字列除去 → 区切りが曖昧で壊れやすい。勤務区分を自由入力 → 表記揺れで集計しづらい。

## Risks / Trade-offs

- **送信完了を検知できない** → ステータスを「メール作成済み」に限定し、送信は自己申告チェックで補助。アプリが送信保証しないことを UI で明示。
- **mailto URL が長すぎると起動失敗** → メモ欄に文字数制限を設け、本文を簡潔に保つ。コピー用フォールバックUIを常設。
- **Outlook が既定メールアプリでない/未インストール** → コピー導線と案内文を表示し、手動送信を可能にする。
- **From が会社アドレスでない可能性** → 確認画面で「Fromが会社メールアドレスか送信前に確認」を促す（From は mailto で指定不可）。
- **添付不可** → 添付はスコープ外と明記。
- **認証なし** → 単一ユーザー・ローカル/限定環境での利用前提。公開ネットワークに晒さない運用とし、必要になった時点で別changeで認証を追加する。
- **Docker から `mailto:` を扱えない** → `mailto:` はコンテナ内ではなくユーザーのブラウザ（ホスト）側で実行される。Docker 化されるのは配信するフロント資産・API・DB であり、Outlook 起動はホストOS上で完結する。

## Migration Plan

新規アプリ（グリーンフィールド）のため移行データなし。デプロイ手順:
1. `docker-compose` で PostgreSQL・FastAPI・フロントの各コンテナを起動
2. Alembic マイグレーションで `work_reports`（`end_mail_body` 含む）・`mail_settings` を作成
3. 設定画面で宛先・テンプレートを登録
4. 記録機能 → メール生成 → mailto 起動 → フォールバックUI → カレンダー表示 の順に段階的にリリース可能

ロールバック: 機能はUI追加が主体で破壊的変更がないため、問題時は該当画面/テーブルを無効化して切り戻す。スキーマ変更は Alembic の downgrade で戻す。
