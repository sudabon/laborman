## Why

会社のMicrosoft 365メールを使った日々の始業・終業報告は、毎日2回・固定宛先への定型メールを手入力する負担が大きい。一方で、Microsoft Graph API や SMTP AUTH による自動送信は M365 管理者承認・権限付与・認証情報管理が必要で、労務報告用途にはコストとリスクが見合わない。Webアプリで報告内容を作成し `mailto:` で Mac の Outlook 起動までを担うことで、管理者承認なしに入力負担だけを大きく削減できる。

## What Changes

- Webアプリで始業・終業を記録し、日付ごとに勤務時間とメール作成状況を管理できるようにする
- 月単位のカレンダーで報告状況（未報告・始業記録済み・終業記録済み・メール作成済み）を可視化する
- 宛先（上司・労務ML・CC・BCC）とメール件名/本文テンプレートを設定画面で管理できるようにする
- テンプレート変数を置換して件名・本文を生成し、`mailto:` リンクで Outlook の新規メール作成画面を開く
- Outlook が起動できない/URLが長すぎる場合に備え、宛先・件名・本文をコピーできるフォールバックUIを提供する
- 送信完了は検知できないため、ステータスは「送信済み」ではなく「メール作成済み」とし、ユーザー自己申告の「Outlookで送信しました」チェックを設ける
- 終業メールのヘッダ（定型挨拶）・フッタ（定型結び）はテンプレートとして PostgreSQL に保存し、設定画面で編集できる（低頻度更新）。日次レコードにはヘッダ/フッタを除いた本文部分のみを保存する。始業メール本文は保存しない
- 勤務区分は「オフィス」「リモート」の2択（固定）とする
- 認証・マルチユーザー対応は行わない（単一ユーザー前提）
- **非対象（このchangeでは扱わない）**: Microsoft Graph API連携、SMTP送信、完全自動送信、送信済み確認、添付ファイル送信、承認ワークフロー、複数ユーザー集計、認証/ログイン

## Capabilities

### New Capabilities
- `work-report-logging`: 始業・終業時刻の記録、当日記録の更新、勤務時間計算、報告ステータス（not_started / start_recorded / start_mail_created / end_recorded / end_mail_created）の管理
- `report-calendar`: 月単位カレンダーでの報告状況表示、未報告日の強調、本日の報告画面からの記録操作
- `mail-settings`: 宛先（上司・労務ML・CC・BCC）と始業/終業の件名・本文テンプレート、終業メールのヘッダ/フッタテンプレートの保存・編集、未設定時の記録操作ガード
- `outlook-mail-compose`: テンプレート変数の置換による件名・本文生成、`mailto:` URL生成、Outlook起動、確認画面、コピー用フォールバックUI

## Impact

- **新規データモデル**: `work_reports`（勤務日・始終業時刻・メール作成日時・メモ・勤務区分〔オフィス/リモート〕・終業メール本文〔ヘッダ/フッタ除く〕）、`mail_settings`（宛先各種・件名/本文テンプレート＋終業ヘッダ/終業フッタテンプレート）。単一ユーザー前提でユーザー単位のスコープは持たない
- **フロントエンド**: カレンダー画面・本日の報告画面・設定画面の追加。スタックは React + TypeScript + Tailwind CSS + shadcn/ui、カレンダーは FullCalendar または React DayPicker
- **バックエンド/永続化**: FastAPI + SQLAlchemy + Alembic、DB は PostgreSQL
- **実行環境**: フロントエンド・バックエンド・PostgreSQL をすべて Docker コンテナとして `docker-compose` で起動する
- **認証**: 不要（単一ユーザー・ローカル/限定環境前提）。マルチユーザー対応も行わない
- **保存しない情報**: M365パスワード・Outlook認証情報・Graphアクセストークン・SMTP/アプリパスワード（セキュリティ上保持しない）、および始業メール本文
- **既知の制約**: `mailto:` 方式では送信完了を検知不可、本文長制限あり、添付不可、From は Outlook 既定アカウント依存
